Dentistry Automation Platform

Production-capable MVP architecture for dental Revenue Cycle Management (RCM). This document explains the full implementation — how eligibility, claims, ERA posting, denials, and exception queues are orchestrated on top of Dentistry Automation's existing DentalBridge PMS layer.

.NET 8 + TypeScript + Python Modular Monolith Mock-First Integrations AI + Rules Hybrid

1. Executive Summary

What this MVP adds to Dentistry Automation's existing products: Dentistry Automation already ships DentalBridge API (PMS-agnostic read) and RCM write-back endpoints (/rcm/eligibility-verification, /rcm/patientpaymentswriteback). This platform adds the missing workflow orchestration layer — stateful RCM pipelines, clearinghouse/EDI routing, acknowledgment monitoring, AI-assisted exception handling, and a partner-facing DA API v1.

Business Goal

Reduce manual RCM work, accelerate cash flow, and let DSOs scale without proportional hiring — sitting directly in the money path.

Architecture Style

Modular monolith (.NET 8) + background workers + Python AI sidecar. Fast iteration for a small team; correctness-critical payment posting stays in deterministic .NET code.

MVP Wedge

ERA/835 posting + eligibility orchestration — highest cash impact, demonstrates AI (EOB extraction) + rules (posting match) hybrid, closes loop with existing PMS write-back.

Integration Strategy

Mock-first adapter ports — full E2E demo today; swap to live Stedi/DentalBridge/production write-back without changing domain logic.

3. Problem → Solution (Hybrid Automation Model)

Current State — High Volume, People Dependent

  • Manual workflows: claim submission, payment posting, claims monitoring, QA/rework, scrubbing, EOB review, secondary follow-up
  • Data silos: PMS, Clearinghouse, Payer Portals, AR Reports — no canonical model
  • Silent 999/277CA failures — claims never adjudicate
  • ERA lines posted manually, line-by-line
  • Denial patterns invisible across DSO locations

Future State — RCM Hybrid Automation Platform (This MVP)

  • Clinician team triggers eligibility checks pre-visit
  • Central platform integrates PMS (DentalBridge), Clearinghouse, Payer data
  • Rules engine handles 80%+ volume automatically
  • RCM team (internal + outsourced) handles exceptions only via work queues
  • AI assists extraction and triage — never auto-moves money below confidence threshold

11. AI Architecture — Rules for Money, AI for Variation

Deterministic (Rules Engine)

  • X12 request formatting
  • CDT / tooth / required-field validation
  • ERA line matching with amount tolerance
  • Auto-post gate (confidence ≥ 0.95)
  • CARC → work queue routing
  • Idempotent payment write-back

AI-Assisted (Python Worker)

  • EOB PDF / text extraction → structured lines
  • Denial summarization (CARC + context)
  • Suggested next action per denial type
  • Work queue priority scoring
  • Benefit summary for clinicians (future)

Human-in-the-Loop Thresholds

OperationAuto ThresholdBelow Threshold
ERA line match + post≥ 0.95 confidenceException work queue
EOB field extraction≥ 0.90 per fieldManual review
Eligibility "Verified"271 only, ≥ 0.95NeedsReview — never auto-verify
Denial recommendationsNever autoHuman must approve before resubmit

AI worker endpoints: POST /extract/eob, POST /summarize/denial, GET /health. Falls back to MockAiWorkerClient if worker is unavailable — platform stays operational.

12. Security & HIPAA Baseline

Authentication

API keys (SHA-256 hashed in DB). Entra ID for staff UI (Phase 2). Demo key for development only.

Tenant Isolation

Every query scoped by OrganizationId. No cross-tenant data access possible at ORM layer.

Audit Trail

Append-only audit_events — who accessed what PHI entity, when, from where.

Encryption

TLS 1.2+ in transit. Azure SQL TDE + Blob SSE at rest. Key Vault for secrets in production.

Rate Limiting

100 requests/minute per IP on /v1/*. Returns HTTP 429 when exceeded.

AI PHI Governance

No training on tenant data. Inference logged with correlation ID. BAA required with AI provider.

Platform is designed as a HIPAA Business Associate. BAA with each practice required before production PHI flows. SOC 2 Type II targeted post-MVP.

17. Architecture Decision Records

DecisionChoiceWhy Not Alternatives
Architecture styleModular monolith + workersMicroservices: too much overhead for small team. Pure monolith: can't scale AI/EDI independently.
PMS integrationDentalBridge read + DA write-backBuilding per-PMS adapters: 6–12 months. DA already has this moat.
ClearinghouseStedi-shaped port (mock first)Direct payer: impossible at scale. DentalXChange only: single point of failure.
DatabaseAzure SQL + EF CoreCosmos: wrong consistency model for money path. Postgres: team .NET/SQL expertise.
AI runtimePython sidecarAll in .NET: weak PDF/ML ecosystem. External SaaS only: PHI governance risk.
Auto-posting gateConfidence ≥ 0.95Auto-post everything: trust risk. Manual everything: no automation value.
